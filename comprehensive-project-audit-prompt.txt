## 🔬 ДЕТАЛЬНЫЙ РАЗБОР ПО КАЖДОМУ БЛОКУ

### 1. Архитектура и структура кода — **68/100**
✅ Разделение на `handlers/`, `services/`, `db/` — хорошо.  
✅ Единый вход (Telegram Bot), монолит, но для MVP нормально.  
✅ Поддержка data scoping по `owner_tg_id` — готовность к multi-user.  

❌ **Слабые места:**
- Нет dependency injection — жёсткая связанность с aiogram.
- FSM-хранилище не указано (Redis / Memory / File?) — риск потери данных при рестарте.
- In-memory `_analysis_in_progress` не переживает рестарт бота (потеря блокировок).
- Отсутствует абстракция над LLM-провайдерами (хардкод Anthropic/OpenAI).
- Нет асинхронного event bus для внутренней коммуникации (напоминания — polling БД).

💥 **Худший сценарий:** Рестарт во время 10 параллельных AI-анализов → потеря idempotency-слотов → двойные запросы к LLM, превышение лимита.

### 2. Логика бота и сценарии (FSM) — **62/100**
✅ Чёткие состояния: `waiting_city`, `waiting_category`, `browsing`, `waiting_note`.  
✅ Команда `/start` сбрасывает FSM.  
✅ Пагинация с индексом 1/42 — корректно.  

❌ **Слабые места:**
- Не описана обработка «cancel» в любой момент (пользователь может отправить `/start`, но это стирает только FSM, а не данные).  
- Нет таймаута FSM-сессии (если пользователь начал ввод города и замолчал на час → FSM висит, следующий /start сбросит, но потеря введённого).
- В состоянии `waiting_note` не указано, как выйти без отправки заметки.
- Поведение при вводе города из >100 символов — просто отсечка, без явного сообщения об ошибке.
- В состоянии `waiting_category` нет кнопки «Назад» для смены города.

💥 **Худший сценарий:** Пользователь начал поиск → ушёл → вернулся через 6 часов → FSM всё ещё активна (если нет таймаута) → при вводе текста попадает в неверное состояние.

### 3. Работа с данными и БД — **58/100**
✅ Дедупликация лидов по `name+address`.  
✅ Ведение лога LLM-вызовов (`llm_call_log`).  
✅ Ограничение выдачи 1000 записей при экспорте (защита от OOM).  

❌ **Слабые места:**
- **⚠️ Отсутствие транзакций:** Вставка лида и запись в `llm_call_log` — отдельные запросы. При падении после INSERT лида и до INSERT лога потеряется учёт вызова → возможен перерасход лимита.
- Нет упоминания индексов (поиск по `owner_tg_id`, `status`, `chat_user_id+chat_message_id` для дедупликации Chat Monitor).
- N+1 неявный: при пагинации лидов (список) для каждого лида может подгружаться дополнительная информация отдельными запросами.
- **Тип БД не указан** — различие SQLite vs PostgreSQL критично для блокировок в конкурентном доступе.
- Напоминания выбираются без `FOR UPDATE` — возможны дубликаты отправки, если два инстанса бота.

💥 **Худший сценарий:** Конкурентная вставка лида (два нажатия «Сохранить» до дедупликации) → дубликаты; при экспорте CSV без лимита 1000 — OOM для 10 000 лидов (ограничение есть, но 1000 это мало).

### 4. Безопасность — **45/100 (КРИТИЧЕСКИ НИЗКО)**
✅ `.env` для токенов.  
✅ Санитизация ввода для Overpass QL (удаление спецсимволов).  
✅ Data scoping по `owner_tg_id`.  
✅ Rate limiting LLM-вызовов (дневной лимит).  

❌ **Критические проблемы:**
- **🔴 Отсутствует rate limiting на входящие сообщения** — злоумышленник может флудить бота сообщениями (каждое обрабатывается, утилизирует ресурсы).
- **🔴 Chat Monitor** — API ID/API Hash хранятся в `.env`, сессия Telethon на диске — если сервер скомпрометирован, злоумышленник получит доступ к аккаунту Telegram.
- **🔴 Персональные данные:** телефоны, имена, сообщения хранятся без шифрования (не упомянуто). GDPR/152-ФЗ — требуется согласие и безопасность.
- **🔴 SQL-инъекции:** не указано использование параметризованных запросов (предположительно есть, но нет гарантии — в описании есть конкатенация в `find_lead_by_name_address`?).
- **🔴 Проверка `exp_user_id` при экспорте** — callback_data содержит `user_id` для проверки. Любой пользователь с доступом к перехваченному callback может подставить чужой ID (если секрет не подписан).
- **Отсутствует аутентификация по ролям** — allowlist в коде? Не описан.

💥 **Худший сценарий:** Утечка `chat_monitor.session` → доступ к аккаунту Telegram владельца → чтение всех переписок, отправка сообщений от его имени.

### 5. Обработка ошибок и устойчивость — **52/100**
✅ Retry напоминаний с откатом `is_sent=False`.  
✅ Fallback при отправке сообщений (HTML → plain).  
✅ Таймаут 30/60 сек на внешние запросы.  

❌ **Слабые места:**
- **Нет глобального try/catch в хендлерах** — любая необработанная ошибка приведёт к падению хендлера (aiogram отправит «Something went wrong», но FSM состояние может быть повреждено).
- **Нет graceful degradation при падении LLM API** — если AI-анализ падает с ошибкой, лид остаётся без анализа, пользователь не получает альтернативы.
- **Нет Health Check** — невозможно понять, что бот жив.
- **Ошибки сети Telegram** (таймаут отправки) могут привести к потере сообщений без уведомления.
- Отсутствует обработка переполнения FSMContext (хранилище может быть переполнено).

💥 **Худший сценарий:** Ошибка в `services/places.py` (Overpass API вернул неожиданный формат) → исключение в хендлере → FSM остается в состоянии `browsing` с частичными данными → пользователь не может выйти.

### 6. Производительность и масштаб — **50/100**
✅ Кэширование OSM-результатов (30 мин).  
✅ Лимит 60 компаний на запрос (разумно).  

❌ **Слабые места:**
- **N+1 в пагинации лидов:** сперва SELECT списка ID, потом для каждого SELECT полной записи или рендеринг карточки.
- **In-memory idempotency set** не масштабируется на несколько экземпляров бота (если будет horizontal scaling).
- **FSM в памяти** (по умолчанию MemoryStorage) — при рестарте все состояния потеряны.
- **Overpass API** — может тормозить или давать ошибки при частых запросах; rate limit OSM не учтён.
- **Чат-монитор** — синхронная обработка сообщений (Telethon on message), может накапливаться очередь.
- **Нет асинхронного worker pool** для фоновых задач (напоминания, LLM-запросы блокируют).

💥 **Худший сценарий:** При 100 пользователях, одновременно запускающих поиск, Overpass API начнёт отклонять запросы, а in-memory кэш будет переполнен.

## 🚨 ТОП-10 КРИТИЧЕСКИХ ПРОБЛЕМ

| № | Проблема | Риск | Влияние | Рекомендация | Трудозатраты |
|---|----------|------|---------|--------------|--------------|
| 1 | **Отсутствие rate limiting на входящие сообщения** | Атака флудом или DoS | Полный отказ бота, перерасход LLM-лимита | Добавить middleware aiogram с лимитом N msg/сек на пользователя | S |
| 2 | **Chat Monitor — уязвимость сессии Telethon** | Утечка API ID/сессии → взлом аккаунта Telegram | Полная компрометация аккаунта владельца | Хранить сессию в зашифрованном виде, использовать отдельный аккаунт | M |
| 3 | **Отсутствие транзакций в БД** | Потеря данных при сбое | Дублирование лидов, рассинхронизация логов | Обернуть в транзакции все группы INSERT/UPDATE | S |
| 4 | **N+1 запросы в пагинации лидов** | Медленная работа при росте данных | Таймауты, плохой UX при >100 лидов | Использовать JOIN или subquery для полной загрузки | M |
| 5 | **Нет тестирования** | Каждая правка может сломать проект | Иллюзия стабильности, частые регрессии | Покрыть unit-тестами services и репозитории, integration для базовых сценариев | L |
| 6 | **FSMMemoryStorage — потеря состояний при рестарте** | Пользователь сбрасывается, введённые данные теряются | Раздражение, повторный ввод | Перейти на Redis FSM storage | M |
| 7 | **In-memory idempotency блокировки** | При горизонтальном масштабировании — дублирующие вызовы LLM | Превышение лимита, двойные траты | Перейти на Redis lock или БД блокировку | M |
| 8 | **Отсутствие мониторинга и алертинга** | Падение ночью — узнается утром | Потеря лидов, снижение доверия | Добавить healthcheck, интеграцию с Sentry/логгером | S |
| 9 | **Нет параметризованных SQL-запросов (не подтверждено)** | SQL-инъекция в полях ввода (город, заметка) | Утечка данных, деструктивные запросы | Использовать ORM (SQLAlchemy) или параметризацию явно | M |
| 10 | **Чат-монитор запускается отдельным процессом** | Сложность оркестрации, может упасть незаметно | Потеря лидов из чатов | Сделать воркер в том же процессе через asyncio или supervisor | L |

## 🎯 ВЕРДИКТ

**НЕТ, этот проект нельзя выкатывать в продакшен в текущем виде.**

Проект представляет собой **хорошо продуманный прототип** с детальным описанием бизнес-логики и пользовательских сценариев. Архитектурные основы (data scoping, дедупликация, лимитирование LLM) заложены правильно. Однако **критические дыры в безопасности, отсутствие тестирования, обработки ошибок и мониторинга** делают его непригодным для реального использования.

Приоритет — закрыть первые 5 проблем из ТОП-10 (🔴 и 🟡), после чего можно рассмотреть soft launch на ограниченной аудитории. После закрытия хотя бы 8 из 10 проблем оценка поднимется до ~65–70, что уже приемлемо для MVP.
